## 1. 개요

### 1.1 애플리케이션 목적

**todolist-app**은 개인 사용자가 자신의 할일을 생성·관리·추적할 수 있는 할일 관리 애플리케이션이다. 인증된 사용자만 자신의 데이터에 접근할 수 있어 개인 정보가 철저히 분리된다.

### 1.2 핵심 문제

| 문제 | 해결 방향 | 관련 UC |
|------|-----------|---------|
| 타인의 할일과 섞여 나만의 목록 관리가 어렵다 | 사용자 인증 기반 개인 데이터 격리 | UC-01, UC-02 |
| 할일의 마감일(종료일)을 지정하는 수단이 없다 | 시작일·종료일 필드 필수화 | UC-03, UC-06 |
| 진행 상황을 한눈에 파악하기 어렵다 | 상태 기반 필터링 및 날짜 정렬 | UC-04 |

### 1.3 타겟 사용자

| 항목 | 내용 |
|------|------|
| 대상 | 10대~50대 일반인 누구나 |
| 등록 사용자 규모 | 10만 명 |
| 동시 접속 사용자 | 최대 500명 |
| 지원 플랫폼 | 모바일 웹, 데스크탑 웹 (반응형 UI) |

### 1.4 기술 스택

| 계층 | 기술 |

## 2. 도메인 개념 및 용어

| 용어 | 정의 |
|------|------|
| **회원(User)** | 회원 가입 후 인증된 서비스 이용자 |
| **할일(Todo)** | 회원이 등록한 단위 과업. 반드시 시작일과 종료일을 가진다 |
| **시작일(startDate)** | 할일을 시작하기로 계획한 날짜 (UTC 기준, `YYYY-MM-DD`) |
| **종료일(dueDate)** | 할일을 완료해야 하는 기한 날짜 (UTC 기준, `YYYY-MM-DD`) |
| **상태(Status)** | 서버의 현재 날짜(UTC)와 시작일·종료일·완료 여부에 따라 결정되는 할일의 진행 단계 |
| **완료 여부(done)** | 회원이 명시적으로 할일을 완수했다고 표시한 플래그 |
| **현재 날짜** | 상태 산출 기준. **서버 UTC 날짜(`YYYY-MM-DD`)**를 사용한다. 모든 날짜 비교는 날짜 수준(일 단위)으로 수행한다 |

## 4. 비즈니스 규칙

### 4.1 회원 규칙

| ID | 규칙 |
|----|------|
| BR-U01 | 이메일은 시스템 전체에서 고유해야 한다 |
| BR-U02 | 비밀번호는 저장 전 반드시 bcrypt(cost factor 12)로 단방향 해시 처리한다 |
| BR-U05 | 비밀번호 길이는 최소 8자, 최대 20자이어야 한다 |
| BR-U06 | 비밀번호는 영문(대소문자), 숫자, 특수문자(`!@#$%^&*-_`) 중 2가지 이상을 조합해야 한다 |
| BR-U03 | 인증되지 않은 사용자는 어떤 할일 데이터에도 접근할 수 없다 |
| BR-U04 | 회원은 자신의 할일만 조회·수정·삭제할 수 있다 |

### 4.2 인증 방식

| ID | 규칙 |
|----|------|
| BR-A01 | 인증은 JWT(HS256 알고리즘) Bearer 토큰 방식, REST API로 제공한다. 토큰 서명에는 서버 환경변수로 관리되는 시크릿 키를 사용한다 |
| BR-A02 | 액세스 토큰 유효 기간은 1시간이다 |
| BR-A03 | 토큰 갱신은 리프레시 토큰(유효 기간 7일)으로 처리한다 |
| BR-A04 | 로그아웃 시 리프레시 토큰은 서버에서 무효화한다 |
| BR-A05 | 만료된 리프레시 토큰으로 갱신 시도 시 `AUTH_004`를 반환하고 재로그인을 요구한다 |

### 4.3 할일 규칙

| ID | 규칙 |
|----|------|
| BR-T01 | 시작일과 종료일은 필수 입력값이다 |
| BR-T02 | 종료일은 시작일과 같거나 이후여야 한다 |
| BR-T03 | `done = true` 설정은 회원이 명시적으로 완료 처리한 경우에만 허용된다 |
| BR-T04 | `done = true → false` 전환(완료 취소)은 허용된다. 상태는 취소 시점의 날짜 기준으로 재계산된다 |

### 4.4 상태(Status) 산출 규칙

상태는 저장 값이 아닌 계산 값이다. 모든 날짜 비교는 **서버 UTC 날짜(일 단위)** 기준으로 수행한다.

| ID | 우선순위 | 조건 | 상태 |
|----|----------|------|------|
| BR-S01 | 1 | `done = true` AND 현재 날짜 ≤ dueDate | **성공완료(Completed)** |
| BR-S02 | 2 | `done = true` AND 현재 날짜 > dueDate | **지연완료(LateCompleted)** |
| BR-S03 | 3 | `done = false` AND 현재 날짜 < startDate | **시작전(Pending)** |
| BR-S04 | 4 | `done = false` AND startDate ≤ 현재 날짜 ≤ dueDate | **진행중(InProgress)** |
| BR-S05 | 5 | `done = false` AND 현재 날짜 > dueDate | **기한초과(Overdue)** |

> 상태 값은 조회 시점에 동적으로 계산되어 응답에 포함된다.

**경계값 검증 케이스**

| 케이스 | done | 현재 날짜 조건 | 기대 상태 |
|--------|------|--------------|-----------|
| 시작일 전날, 미완료 | false | 현재 < startDate | 시작전 |
| 시작일 당일, 미완료 | false | 현재 = startDate | 진행중 |
| 종료일 당일, 미완료 | false | 현재 = dueDate | 진행중 |
| 종료일 다음날, 미완료 | false | 현재 = dueDate + 1일 | 기한초과 |
| 종료일 당일, 완료 | true | 현재 = dueDate | 성공완료 |
| 종료일 다음날, 완료 | true | 현재 = dueDate + 1일 | 지연완료 |
| startDate = dueDate, 미완료, 당일 | false | 현재 = startDate = dueDate | 진행중 |

### 4.5 정렬 규칙

| ID | 규칙 |
|----|------|
| BR-O01 | 기본 정렬: 종료일(dueDate) 오름차순 |
| BR-O02 | 지원 정렬 기준: 시작일(startDate), 종료일(dueDate), 생성일(createdAt) |
| BR-O03 | 정렬 방향: 오름차순(ASC) / 내림차순(DESC) |
| BR-O04 | 동일 정렬 기준값을 가진 항목은 createdAt 오름차순으로 처리한다 (동률 처리) |

## 5. 오류 코드 정의

| 코드 | HTTP 상태 | 설명 |
|------|-----------|------|

## 7. 주요 유스케이스

### UC-01: 회원 가입

- **행위자**: 비인증 사용자
- **사전 조건**: 해당 이메일로 가입된 계정이 없음
- **흐름**: 이메일·비밀번호·이름 입력 → 유효성 검증 → 계정 생성 → 로그인 페이지로 이동
- **예외**: 중복 이메일 → `AUTH_003`, 비밀번호 정책 미충족 → `USER_001`
- **관련 규칙**: BR-U01, BR-U02, BR-U05, BR-U06, NFR-005

### UC-02: 로그인

- **행위자**: 가입된 회원
- **흐름**: 이메일·비밀번호 입력 → 인증 → 액세스 토큰 및 리프레시 토큰 발급
- **예외**: 자격 증명 불일치 → `AUTH_002`
- **관련 규칙**: BR-A01, BR-A02, BR-A03

### UC-03: 할일 생성

- **행위자**: 인증된 회원
- **사전 조건**: 유효한 액세스 토큰 보유
- **흐름**: 제목·시작일·종료일 입력(설명 선택) → 유효성 검증 → 저장 → 생성된 할일 반환
- **예외**: 필수 필드 누락 → `TODO_001`, 종료일 < 시작일 → `TODO_002`, 토큰 없음/유효하지 않음 → `AUTH_001`
- **관련 규칙**: BR-T01, BR-T02, BR-U03

### UC-04: 할일 목록 조회 (필터·정렬·페이지네이션)

- **행위자**: 인증된 회원
- **사전 조건**: 유효한 액세스 토큰 보유
- **흐름**: 상태 필터(선택)·정렬 기준(선택)·페이지(선택) 지정 → 본인 할일만 조회 → 상태 계산 후 페이지 단위로 반환
- **지원 필터**: `진행중`, `기한초과`, `시작전`, `성공완료`, `지연완료` (복수 선택 가능)
- **지원 정렬**: 시작일, 종료일, 생성일 기준 오름차순/내림차순
- **페이지네이션**: 기본 20건, 최대 100건 (→ NFR-003)
- **관련 규칙**: BR-S01~BR-S05, BR-O01~BR-O04, BR-U04, NFR-003

### UC-05: 할일 완료 처리 / 완료 취소

- **행위자**: 인증된 회원
- **사전 조건**: 유효한 액세스 토큰 보유
- **흐름**: 특정 할일 선택 → `done` 값 토글(true ↔ false) → 상태 재계산 → 업데이트된 할일 반환
- **예외**: 타인 소유 할일 → `TODO_003`, 존재하지 않는 할일 → `TODO_004`
- **관련 규칙**: BR-T03, BR-T04, BR-S01~BR-S05, BR-U04

### UC-06: 할일 수정

- **행위자**: 인증된 회원
- **사전 조건**: 유효한 액세스 토큰 보유
- **흐름**: 특정 할일 선택 → 수정 가능 필드(제목·설명·시작일·종료일) 변경 → 유효성 검증 → 저장
- **예외**: 종료일 < 시작일 → `TODO_002`, 타인 소유 할일 → `TODO_003`
- **관련 규칙**: BR-T01, BR-T02, BR-U04

### UC-07: 할일 삭제

- **행위자**: 인증된 회원
- **사전 조건**: 유효한 액세스 토큰 보유
- **흐름**: 특정 할일 선택 → 소유 확인 → 삭제
- **예외**: 타인 소유 할일 → `TODO_003`, 존재하지 않는 할일 → `TODO_004`
- **관련 규칙**: BR-U04

### UC-08: 로그아웃

- **행위자**: 인증된 회원
- **흐름**: 로그아웃 요청 → 서버에서 리프레시 토큰 무효화 → 클라이언트 토큰 삭제
- **관련 규칙**: BR-A04

### UC-09: 회원 정보 수정

- **행위자**: 인증된 회원
- **사전 조건**: 유효한 액세스 토큰 보유
- **흐름**: 수정할 필드(이름, 비밀번호) 입력 → 현재 비밀번호 재확인(비밀번호 변경 시) → 유효성 검증 → 저장
- **예외**: 비밀번호 정책 미충족 → `USER_001`, 현재 비밀번호 불일치 → `AUTH_002`
- **관련 규칙**: BR-U02, BR-U05, BR-U06, NFR-005
