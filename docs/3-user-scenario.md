### SCN-01: 신규 사용자 가입 및 첫 할일 등록 (해피 패스)

- **시나리오 ID**: SCN-01
- **페르소나**: P-B — 대학생 이수연 (22세). 스마트폰으로 이동 중에 앱을 처음 접한다. 다음 주 제출해야 하는 과제를 잊기 전에 빠르게 등록하고 싶다.
- **목표**: 신규 계정을 만들고, 첫 번째 할일("경제학 기말 보고서")을 시작일/종료일과 함께 등록한다.
- **전제 조건**:
  - 이수연은 서비스에 등록된 계정이 없다.
  - 브라우저에서 `/register` 화면에 접속한 상태다.
- **시나리오 흐름**:
  1. 이수연은 회원 가입 화면(SCR-02)에서 이름 "이수연", 이메일 "suyeon@uni.ac.kr", 비밀번호 "Study2026!" 을 입력한다.
  2. 비밀번호 확인 필드에도 동일하게 "Study2026!" 을 입력하고 가입 버튼을 클릭한다.
  3. 시스템은 `POST /api/auth/register`를 호출한다. 이메일 중복 여부를 확인하고(BR-U01), 비밀번호 정책(BR-U05, BR-U06)을 검증한다.
  4. 검증을 통과하면 시스템이 계정을 생성하고, 이수연을 로그인 화면(SCR-01)으로 이동시킨다.
  5. 이수연은 방금 가입한 이메일과 비밀번호를 입력하고 로그인 버튼을 누른다.
  6. 시스템은 `POST /api/auth/login`을 호출해 자격 증명을 검증하고, 액세스 토큰(유효 1시간)과 리프레시 토큰(유효 7일)을 발급한다(BR-A01, BR-A02, BR-A03).
  7. 이수연은 할일 목록 화면(SCR-03, `/`)으로 이동한다. 아직 등록된 할일이 없으므로 빈 목록이 표시된다.
  8. 이수연은 "새 할일 추가" 버튼을 누른다.
  9. 할일 생성 화면(SCR-04, `/todos/new`)에서 제목 "경제학 기말 보고서", 시작일 "2026-04-03", 종료일 "2026-04-10" 을 입력한다. 설명 필드는 비워 둔다.
  10. 저장 버튼을 클릭하면 시스템이 `POST /api/todos`를 호출한다. 종료일(04-10) >= 시작일(04-03) 조건을 만족하므로(BR-T02) 할일이 저장된다.
  11. 현재 날짜(2026-04-01)는 시작일(04-03)보다 이르므로 시스템은 상태를 **시작전(Pending)** 으로 계산한다(BR-S03).
  12. 이수연은 할일 목록 화면으로 돌아오고, 방금 등록한 "경제학 기말 보고서" 카드가 "시작전" 배지와 함께 표시된다.
- **결과**: 이수연의 계정이 생성되고, 첫 번째 할일이 "시작전" 상태로 목록에 등록된다.
- **관련 UC**: UC-01, UC-02, UC-03, UC-04

### SCN-02: 중복 이메일로 가입 시도 (예외)

- **시나리오 ID**: SCN-02
- **페르소나**: P-B — 대학생 이수연 (22세). 이전에 가입했던 사실을 잊고 같은 이메일로 다시 가입을 시도한다.
- **목표**: 중복 이메일 오류 메시지를 확인하고 로그인 화면으로 이동한다.
- **전제 조건**:
  - "suyeon@uni.ac.kr" 이메일로 이미 가입된 계정이 존재한다.
  - 이수연은 `/register` 화면에 접속한 상태다.
- **시나리오 흐름**:
  1. 이수연은 회원 가입 화면(SCR-02)에서 이름 "이수연 2", 이메일 "suyeon@uni.ac.kr", 비밀번호 "NewPass9#" 을 입력하고 가입 버튼을 클릭한다.
  2. 시스템은 `POST /api/auth/register`를 호출한다. 이메일 중복 여부를 확인하고 이미 등록된 이메일임을 감지한다(BR-U01).
  3. 시스템은 오류 코드 `AUTH_003`(HTTP 409)을 반환한다.

  6. 이수연은 로그인 화면(SCR-01)으로 이동한다.
- **결과**: 계정이 추가로 생성되지 않는다. 이수연은 오류 메시지를 통해 이미 가입된 이메일임을 인지하고 로그인 화면으로 이동한다.
- **관련 UC**: UC-01

### SCN-03: 비밀번호 정책 위반으로 가입 실패 (예외)

- **시나리오 ID**: SCN-03
- **페르소나**: P-B — 대학생 이수연 (22세). 처음 회원 가입을 시도하면서 비밀번호를 단순하게 입력해 오류를 만난다.
- **목표**: 비밀번호 정책 오류 메시지를 확인하고 올바른 비밀번호로 수정하여 가입을 완료한다.
- **전제 조건**:
  - "suyeon2@uni.ac.kr" 이메일로 가입된 계정이 없다.
  - 이수연은 `/register` 화면에 접속한 상태다.
- **시나리오 흐름**:
  1. 이수연은 회원 가입 화면(SCR-02)에서 이름 "이수연", 이메일 "suyeon2@uni.ac.kr", 비밀번호 "1234" 를 입력하고 가입 버튼을 클릭한다.
  2. 시스템은 `POST /api/auth/register`를 호출한다. 비밀번호 길이(4자)가 최소 8자 미만임을 감지한다(BR-U05).
  3. 시스템은 오류 코드 `USER_001`(HTTP 400)을 반환한다.
  4. 화면에 "비밀번호는 8~20자이며, 영문·숫자·특수문자 중 2가지 이상을 조합해야 합니다." 안내 문구가 표시된다.
  5. 이수연은 비밀번호를 "suyeon" 으로 수정하고 다시 가입을 시도한다. 이번엔 길이(6자)는 여전히 미충족이다.
  6. 시스템은 다시 `USER_001`을 반환한다.
  7. 이수연은 화면의 정책 안내 문구를 자세히 읽고 비밀번호를 "Study2026!" 으로 수정한다. 길이 10자, 영문+숫자+특수문자 조합으로 정책을 충족한다(BR-U05, BR-U06).
  8. 시스템은 검증을 통과하고 계정을 생성한다. 이수연은 로그인 화면으로 이동한다.
- **결과**: 처음 두 번의 시도는 `USER_001` 오류로 실패하고, 세 번째 시도에서 정책을 충족한 비밀번호로 계정이 생성된다.
- **관련 UC**: UC-01

### SCN-04: 로그인 후 오늘 마감인 할일 확인 (해피 패스)

- **시나리오 ID**: SCN-04
- **페르소나**: P-A — 직장인 김민준 (32세). 출근 후 데스크탑에서 오늘 마감인 업무 과제를 한눈에 파악하려 한다.
- **목표**: 로그인 후 종료일이 오늘(2026-04-01)인 할일을 빠르게 확인한다.
- **전제 조건**:
  - 김민준은 "minjun@company.com" 계정으로 이미 가입되어 있다.
  - 계정에는 다음 할일들이 등록되어 있다.
    - "주간 보고서 작성": 시작일 2026-03-30, 종료일 2026-04-01, done=false
    - "팀 워크숍 자료 준비": 시작일 2026-04-01, 종료일 2026-04-03, done=false
    - "분기 예산 검토": 시작일 2026-03-25, 종료일 2026-03-31, done=false
  - 현재 서버 UTC 날짜는 2026-04-01이다.
- **시나리오 흐름**:
  1. 김민준은 로그인 화면(SCR-01)에서 이메일과 비밀번호를 입력하고 로그인 버튼을 클릭한다.
  2. 시스템이 `POST /api/auth/login`을 호출해 자격 증명을 확인하고 토큰을 발급한다.
  3. 김민준은 할일 목록 화면(SCR-03)으로 이동한다. 기본 정렬(종료일 오름차순)이 적용된 상태다(BR-O01).
  4. 시스템은 `GET /api/todos`를 호출한다. 현재 날짜 2026-04-01 기준으로 각 할일의 상태를 계산한다.
     - "분기 예산 검토": done=false, 현재(04-01) > dueDate(03-31) → **기한초과(Overdue)** (BR-S05)
     - "주간 보고서 작성": done=false, startDate(03-30) <= 현재(04-01) <= dueDate(04-01) → **진행중(InProgress)** (BR-S04)
     - "팀 워크숍 자료 준비": done=false, startDate(04-01) <= 현재(04-01) <= dueDate(04-03) → **진행중(InProgress)** (BR-S04)
  5. 목록은 종료일 오름차순으로 정렬되어 "분기 예산 검토(기한초과)" → "주간 보고서 작성(진행중)" → "팀 워크숍 자료 준비(진행중)" 순으로 표시된다.
  6. 김민준은 "주간 보고서 작성" 카드의 종료일(2026-04-01)이 오늘임을 확인한다. 해당 카드는 종료일 배지가 강조 표시된다.
  7. 김민준은 가장 위에 표시된 "분기 예산 검토" 가 이미 기한을 넘겼음을 함께 인지하고 우선 처리가 필요하다는 것을 파악한다.
- **결과**: 김민준은 로그인 직후 오늘 마감인 할일("주간 보고서 작성")과 기한초과 할일("분기 예산 검토")을 한눈에 파악한다.
- **관련 UC**: UC-02, UC-04

### SCN-05: 기한초과 할일 필터링 및 완료 처리 (해피 패스)

- **시나리오 ID**: SCN-05
- **페르소나**: P-C — 프리랜서 박지훈 (41세). 여러 클라이언트 프로젝트를 동시에 진행 중이며, 기한이 지난 미완료 할일만 모아서 빠르게 처리하려 한다.
- **목표**: 기한초과 상태의 할일만 필터링해서 조회하고, 완료 처리한다.
- **전제 조건**:
  - 박지훈은 로그인 상태이며 유효한 액세스 토큰을 보유하고 있다.
  - 계정에는 다양한 상태의 할일이 10건 이상 등록되어 있으며, 이 중 기한초과(Overdue) 상태의 할일이 3건 있다.
  - 현재 서버 UTC 날짜는 2026-04-01이다.
- **시나리오 흐름**:
  1. 박지훈은 할일 목록 화면(SCR-03)에서 상태 필터 탭을 확인한다. 현재 "전체" 탭이 선택된 상태로 완료/미완료 항목이 섞여 있다.
  2. 박지훈은 "기한초과" 필터 탭을 클릭한다.
  3. 시스템은 `GET /api/todos?status=Overdue`를 호출한다. 서버는 현재 날짜(2026-04-01) 기준으로 `done=false AND 현재 > dueDate` 조건을 만족하는 항목만 반환한다(BR-S05, AC-04-2).
  4. 화면에 기한초과 할일 3건만 종료일 오름차순으로 표시된다(BR-O01).
  5. 박지훈은 가장 오래된 기한초과 할일인 "A사 로고 시안 제출" 카드를 확인한다. 이미 클라이언트에게 납품을 완료한 건이다.
  6. 박지훈은 해당 카드의 완료 처리 버튼을 클릭한다.
  7. 시스템은 `PATCH /api/todos/{id}/done`을 호출한다. 해당 할일의 `done` 값이 false → true로 변경된다(BR-T03).
  8. 시스템은 현재 날짜(2026-04-01)와 dueDate를 비교한다. `done=true AND 현재 > dueDate` 이므로 상태를 **지연완료(LateCompleted)** 로 계산한다(BR-S02).
  9. "기한초과" 필터가 적용된 상태에서 해당 항목은 Overdue 조건을 벗어났으므로 목록에서 제거된다. 남은 기한초과 할일 2건이 표시된다.
  10. 박지훈은 나머지 2건도 동일하게 완료 처리한다. 기한초과 목록이 비게 된다.
  11. 화면에 "기한초과 항목이 없습니다." 메시지가 표시된다.
- **결과**: 기한초과 상태 3건 모두 완료 처리되어 지연완료(LateCompleted) 상태로 전환된다. 기한초과 필터 목록이 비게 된다.
- **관련 UC**: UC-04, UC-05

### SCN-06: 할일 시작일/종료일 잘못 입력 시 오류 (예외)

- **시나리오 ID**: SCN-06
- **페르소나**: P-A — 직장인 김민준 (32세). 급하게 새 할일을 등록하다가 시작일과 종료일을 실수로 잘못 입력한다.
- **목표**: 날짜 유효성 오류 메시지를 확인하고 올바르게 수정하여 할일을 등록한다.
- **전제 조건**:
  - 김민준은 로그인 상태이며 할일 생성 화면(SCR-04, `/todos/new`)에 진입한 상태다.
- **시나리오 흐름**:
  1. 김민준은 빠르게 할일 생성 화면에서 제목 "고객사 제안서 제출" 을 입력한다.
  2. 시작일 날짜 선택기에서 실수로 시작일을 "2026-04-10", 종료일을 "2026-04-05" 로 선택한다(종료일이 시작일보다 이르다).
  3. 저장 버튼을 클릭한다.
  4. 시스템은 `POST /api/todos`를 호출한다. 종료일(04-05) < 시작일(04-10) 이므로 BR-T02 위반을 감지한다.
  5. 시스템은 오류 코드 `TODO_002`(HTTP 400)을 반환한다.
  6. 화면의 오류 메시지 표시 영역에 "종료일은 시작일과 같거나 이후여야 합니다." 안내가 표시된다.
  7. 김민준은 날짜 선택기에서 종료일을 "2026-04-15" 로 수정한다. 이제 종료일(04-15) >= 시작일(04-10)을 만족한다.
  8. 다시 저장 버튼을 클릭한다. 시스템이 유효성 검증을 통과하고 할일을 저장한다(AC-03-1).
  9. 김민준은 목록 화면으로 돌아오고 새로 등록된 할일이 "시작전(Pending)" 상태로 표시된다.
- **결과**: 잘못된 날짜 입력은 `TODO_002` 오류로 거부되고, 올바른 날짜로 수정 후 할일이 정상 등록된다.
- **관련 UC**: UC-03

### SCN-07: 액세스 토큰 만료 후 자동 갱신 흐름 (엣지 케이스)

- **시나리오 ID**: SCN-07
- **페르소나**: P-C — 프리랜서 박지훈 (41세). 점심 식사 중 자리를 비웠다가 1시간 이상 후에 돌아와 앱을 계속 사용하려 한다.
- **목표**: 액세스 토큰 만료 후 리프레시 토큰으로 자동 갱신하여 사용자가 재로그인 없이 서비스를 이용한다.
- **전제 조건**:
  - 박지훈은 1시간 15분 전에 로그인했으며, 액세스 토큰은 만료된 상태다(BR-A02).
  - 리프레시 토큰은 발급 후 2일이 지났으며 아직 7일 유효 기간 내에 있다(BR-A03).
  - 클라이언트(프론트엔드)는 API 요청 실패 시 토큰 자동 갱신을 시도하는 인터셉터를 구현하고 있다.
- **시나리오 흐름**:
  1. 박지훈은 자리에 돌아와 할일 목록을 새로고침하려 화면을 조작한다.
  2. 클라이언트가 만료된 액세스 토큰을 Authorization 헤더에 포함해 `GET /api/todos`를 호출한다.
  3. 서버는 토큰이 만료됐음을 감지하고 `AUTH_004`(HTTP 401)를 반환한다.
  4. 클라이언트의 인터셉터가 `AUTH_004` 응답을 감지하고 자동으로 `POST /api/auth/refresh`를 호출한다. 저장된 리프레시 토큰을 요청 본문에 포함한다.
  5. 서버는 리프레시 토큰이 유효함을 확인하고(BR-A03), 새 액세스 토큰(유효 1시간)을 발급한다.
  6. 클라이언트는 새 액세스 토큰을 저장하고, 실패했던 원래 요청(`GET /api/todos`)을 새 토큰으로 재시도한다.
  7. 서버가 정상적으로 할일 목록을 반환한다. 박지훈의 화면에 최신 할일 목록이 표시된다.
  8. 박지훈은 토큰이 만료되었다는 사실을 인지하지 못한 채 자연스럽게 서비스를 계속 이용한다.
- **결과**: 액세스 토큰 만료가 사용자에게 노출되지 않고 자동으로 갱신되어 서비스 연속성이 유지된다. 박지훈은 재로그인 없이 작업을 이어간다.
- **관련 UC**: UC-02

> **엣지 케이스 — 리프레시 토큰도 만료된 경우**: 만약 리프레시 토큰도 만료된 상태라면(7일 경과 또는 로그아웃으로 무효화), `POST /api/auth/refresh`에서 `AUTH_005`(HTTP 401)가 반환된다. 클라이언트는 이 경우 로그인 화면(SCR-01)으로 강제 이동시키고 "세션이 만료되었습니다. 다시 로그인해 주세요." 메시지를 표시한다(BR-A05).

### SCN-08: 완료한 할일을 완료 취소하는 흐름 (해피 패스)

- **시나리오 ID**: SCN-08
- **페르소나**: P-A — 직장인 김민준 (32세). 어제 완료 처리했던 할일이 실은 아직 끝나지 않았음을 오늘 아침에 깨달았다.
- **목표**: done=true로 표시된 할일을 완료 취소(done=false)하고, 현재 날짜 기준으로 상태가 재계산되는 것을 확인한다.
- **전제 조건**:
  - 김민준은 로그인 상태이며 유효한 액세스 토큰을 보유하고 있다.
  - "팀 성과 보고서 작성" 할일이 다음 상태로 존재한다: 시작일 2026-03-28, 종료일 2026-04-02, done=true.
  - 어제(2026-03-31) 완료 처리 당시 dueDate(04-02) >= 현재(03-31)이었으므로 상태는 **성공완료(Completed)** 였다.
  - 현재 서버 UTC 날짜는 2026-04-01이다.
- **시나리오 흐름**:
  1. 김민준은 할일 목록 화면(SCR-03)에서 "성공완료" 상태의 "팀 성과 보고서 작성" 카드를 확인한다.
  2. 해당 카드를 클릭해 상세 정보를 확인한다. 보고서 일부가 수정이 필요하다는 것을 깨닫는다.
  3. 카드의 완료 취소 버튼(또는 완료 체크박스 재클릭)을 누른다.
  4. 시스템은 `PATCH /api/todos/{id}/done`을 호출한다. 해당 할일의 `done` 값이 true → false로 변경된다(BR-T04).
  5. 시스템은 취소 시점의 현재 날짜(2026-04-01)를 기준으로 상태를 재계산한다.
     - done=false, startDate(03-28) <= 현재(04-01) <= dueDate(04-02) → **진행중(InProgress)** (BR-S04)
  6. 서버는 업데이트된 할일 객체를 반환한다. 상태는 "진행중" 이다.
  7. 화면에서 해당 카드의 배지가 "성공완료" 에서 "진행중" 으로 변경된다.
  8. 김민준은 보고서 수정을 완료하면 다시 완료 처리를 할 계획임을 인지하고 목록을 닫는다.
- **결과**: 완료 취소가 정상적으로 처리되고, 상태가 현재 날짜 기준 "진행중(InProgress)" 으로 재계산되어 목록에 반영된다.
- **관련 UC**: UC-05

> **참고**: 만약 완료 취소 시점에 종료일이 이미 지났다면(예: 현재 > dueDate), 상태는 **기한초과(Overdue)** 로 재계산된다(BR-S05). 이 경우 김민준은 기한초과 상태로 표시된 카드를 보고 즉시 조치가 필요하다는 것을 인지할 수 있다.

### SCN-09: 회원 탈퇴 흐름 (해피 패스)

- **시나리오 ID**: SCN-09
- **페르소나**: P-B — 대학생 이수연 (22세). 학기가 끝나고 더 이상 앱을 사용하지 않기로 결정했다. 개인 정보가 남지 않도록 계정을 완전히 삭제하고 싶다.
- **목표**: 비밀번호를 재확인하고 계정과 모든 할일 데이터를 영구 삭제한다.
- **전제 조건**:
  - 이수연은 로그인 상태이며 유효한 액세스 토큰을 보유하고 있다.
  - 계정에 5건의 할일 데이터가 존재한다.
- **시나리오 흐름**:
  1. 이수연은 할일 목록 화면(SCR-03) 상단의 "내 정보" 링크를 클릭한다.
  2. 내 정보 화면(SCR-05, `/profile`)으로 이동한다.
  3. 화면 하단의 "회원 탈퇴" 버튼을 클릭한다.
  4. 시스템이 비밀번호 재확인 모달을 표시한다. "탈퇴 시 모든 데이터가 즉시 삭제되며 복구가 불가능합니다. 비밀번호를 입력해 주세요." 경고 문구가 포함되어 있다.
  5. 이수연은 현재 비밀번호 "Study2026!" 을 입력하고 "탈퇴 확인" 버튼을 클릭한다.
  6. 시스템은 `DELETE /api/users/me`를 호출한다. 액세스 토큰의 사용자 ID로 비밀번호를 bcrypt 검증한다(BR-U02).
  7. 비밀번호 확인이 성공하면 시스템은 데이터베이스에서 CASCADE 삭제를 실행한다.
     - `todos` 테이블에서 이수연 소유의 할일 5건이 즉시 삭제된다(NFR-004).
     - `refresh_tokens` 테이블에서 이수연의 리프레시 토큰이 삭제된다.
     - `users` 테이블에서 이수연의 계정 레코드가 삭제된다.
  8. 시스템은 클라이언트에 저장된 액세스 토큰과 리프레시 토큰을 삭제 처리한다.
  9. 이수연은 로그인 화면(SCR-01)으로 리다이렉트된다. "탈퇴가 완료되었습니다." 안내 메시지가 표시된다.
  10. 이수연이 해당 이메일로 다시 로그인을 시도하면 `AUTH_002` 오류가 반환된다(계정 없음).
- **결과**: 이수연의 계정과 모든 할일 데이터(5건)가 즉시 삭제된다. 해당 이메일로의 로그인이 불가능해진다.
- **관련 UC**: UC-10

> **예외 — 비밀번호 불일치**: 5단계에서 잘못된 비밀번호를 입력하면 `AUTH_002`(HTTP 401)가 반환된다. 모달의 오류 메시지 영역에 "비밀번호가 올바르지 않습니다." 가 표시되고 탈퇴는 진행되지 않는다(AC-10-2).

### SCN-10: 다양한 정렬 기준으로 할일 목록 탐색 (해피 패스)

- **시나리오 ID**: SCN-10
- **페르소나**: P-C — 프리랜서 박지훈 (41세). 여러 클라이언트 프로젝트의 할일이 쌓여 있어 상황에 따라 정렬 기준을 바꿔가며 우선순위를 파악한다.
- **목표**: 종료일 오름차순 → 생성일 내림차순 → 시작일 오름차순으로 순서를 바꿔가며 목록을 탐색한다.
- **전제 조건**:
  - 박지훈은 로그인 상태이며 유효한 액세스 토큰을 보유하고 있다.
  - 계정에 다음 할일들이 등록되어 있다.
    - "B사 콘텐츠 기획안": 시작일 2026-04-05, 종료일 2026-04-12, 생성일 2026-03-20
    - "C사 UI 디자인 검토": 시작일 2026-04-01, 종료일 2026-04-08, 생성일 2026-03-28
    - "D사 계약서 검토": 시작일 2026-04-03, 종료일 2026-04-15, 생성일 2026-04-01
    - "A사 최종 납품": 시작일 2026-04-01, 종료일 2026-04-07, 생성일 2026-03-15
- **시나리오 흐름**:
  1. 박지훈은 할일 목록 화면(SCR-03)에 진입한다. 기본 정렬인 "종료일 오름차순(dueDate ASC)" 이 적용된다(BR-O01).
  2. 시스템은 `GET /api/todos?sortBy=dueDate&order=ASC`를 호출한다. 결과:
     - "A사 최종 납품" (04-07) → "C사 UI 디자인 검토" (04-08) → "B사 콘텐츠 기획안" (04-12) → "D사 계약서 검토" (04-15) 순으로 표시된다.
  3. 박지훈은 가장 급한 마감 순서를 확인했다. 이번에는 최근에 추가한 할일이 무엇인지 확인하고 싶다.
  4. 정렬 기준 셀렉트에서 "생성일" 을 선택하고 방향을 "내림차순(DESC)" 으로 설정한다.
  5. 시스템은 `GET /api/todos?sortBy=createdAt&order=DESC`를 호출한다(AC-04-3). 결과:
     - "D사 계약서 검토" (04-01 생성) → "C사 UI 디자인 검토" (03-28 생성) → "B사 콘텐츠 기획안" (03-20 생성) → "A사 최종 납품" (03-15 생성) 순으로 표시된다.
  6. 박지훈은 오늘 막 추가한 "D사 계약서 검토" 가 맨 위에 있음을 확인한다.
  7. 이번에는 언제 시작해야 하는지 순서대로 보고 싶다. 정렬 기준을 "시작일 오름차순(startDate ASC)" 으로 변경한다.
  8. 시스템은 `GET /api/todos?sortBy=startDate&order=ASC`를 호출한다. 결과:
     - "C사 UI 디자인 검토" 와 "A사 최종 납품" 은 동일한 시작일(04-01)이므로 생성일 오름차순(BR-O04)으로 처리되어 "A사 최종 납품"(03-15 생성) → "C사 UI 디자인 검토"(03-28 생성) 순서가 된다.
     - 최종 순서: "A사 최종 납품" → "C사 UI 디자인 검토" → "D사 계약서 검토" (04-03) → "B사 콘텐츠 기획안" (04-05) 순으로 표시된다.
  9. 박지훈은 시작일 기준으로 일정을 파악하고 이번 주 집중할 작업이 2건("A사 최종 납품", "C사 UI 디자인 검토") 임을 확인한다.
- **결과**: 박지훈은 정렬 기준을 세 가지로 바꾸어 가며 마감 순, 최근 추가 순, 시작일 순으로 할일 목록을 탐색한다. 동일 시작일 항목은 생성일 오름차순으로 안정적으로 정렬된다(BR-O04).
- **관련 UC**: UC-04

| 우선순위 | 조건 | 상태 | 관련 시나리오 |
|----------|------|------|---------------|
| 1 | done=true AND 현재 <= dueDate | 성공완료 (Completed) | SCN-08 |
| 2 | done=true AND 현재 > dueDate | 지연완료 (LateCompleted) | SCN-05 |
| 3 | done=false AND 현재 < startDate | 시작전 (Pending) | SCN-01 |
| 4 | done=false AND startDate <= 현재 <= dueDate | 진행중 (InProgress) | SCN-04, SCN-08 |
| 5 | done=false AND 현재 > dueDate | 기한초과 (Overdue) | SCN-04, SCN-05 |
