## 1. 제품 개요

**한 줄 설명**
개인 인증 기반 할일 관리 웹 애플리케이션 — 사용자가 자신의 할일을 생성·추적·완료할 수 있으며, 시작일과 종료일을 기반으로 상태를 자동 산출한다.

**핵심 가치 제안**
- 인증된 사용자만 자신의 데이터에 접근 → 완전한 개인 데이터 격리
- 시작일·종료일 필수화 → 기한 기반 할일 관리
- 서버 UTC 기준 상태 자동 계산 → 별도 조작 없이 진행 상황 파악

## 2. 사용자 및 페르소나

### 페르소나 A — 직장인 김민준 (32세)

- 상황: 매일 여러 업무 과제를 병행하며 마감일 관리가 필요
- 목표: 업무 할일을 종료일 순으로 정렬해 우선순위를 파악
- 주요 이용 패턴: 데스크탑에서 출근 후 할일 목록 확인, 퇴근 전 완료 처리
- 불편 사항: 기존 메모 앱은 마감일 필터가 없어 기한 초과를 자주 놓침

### 페르소나 B — 대학생 이수연 (22세)

- 상황: 과제·시험·동아리 활동 등 일정이 불규칙
- 목표: 시작일과 종료일을 명시해 언제 시작해야 하는지 시각적으로 확인
- 주요 이용 패턴: 모바일 웹으로 이동 중 빠르게 할일 등록 및 상태 확인
- 불편 사항: 할일이 아직 시작할 필요 없는 상태인지(시작전) 직관적으로 구분하기 어려움

### 페르소나 C — 프리랜서 박지훈 (41세)

- 상황: 다수 클라이언트 프로젝트를 동시 진행
- 목표: 기한이 지난 미완료 할일(기한초과)을 빠르게 식별하고 처리
- 주요 이용 패턴: 데스크탑/모바일 혼용, 상태 필터로 기한초과 항목만 조회
- 불편 사항: 완료된 항목과 미완료 항목이 섞여 표시되면 업무 집중도 저하

## 3. 기능 요구사항

### UC-01: 회원 가입

**설명**
비인증 사용자가 이메일·비밀번호·이름으로 계정을 생성한다.

**사용자 스토리**
> As a 비인증 사용자, I want to 이메일과 비밀번호로 계정을 만들고 싶다, So that 나만의 할일 목록을 관리할 수 있다.

**수용 기준 (Given-When-Then)**

| # | Given | When | Then |
|---|-------|------|------|
| AC-01-1 | 이메일이 시스템에 등록되지 않은 상태 | 유효한 이메일·비밀번호(8~20자, 영문·숫자·특수문자 중 2가지 이상 조합)·이름을 입력하고 가입 요청 | 계정이 생성되고 로그인 페이지로 이동한다 |
| AC-01-2 | 이미 등록된 이메일이 존재하는 상태 | 동일 이메일로 가입 요청 | AUTH_003 오류를 반환한다 |
| AC-01-3 | 비밀번호 7자 이하 입력 | 가입 요청 | USER_001 오류를 반환한다 (길이 미충족) |
| AC-01-4 | 비밀번호 21자 이상 입력 | 가입 요청 | USER_001 오류를 반환한다 (길이 초과) |
| AC-01-5 | 비밀번호 8자 이상이나 영문만 입력 | 가입 요청 | USER_001 오류를 반환한다 (복잡도 미충족) |

**사용자 스토리**
> As a 가입된 회원, I want to 이메일과 비밀번호로 로그인하고 싶다, So that 내 할일 데이터에 접근할 수 있다.

**수용 기준**

### UC-03: 할일 생성

**설명**
인증된 회원이 제목·시작일·종료일(+선택 설명)을 입력해 새 할일을 등록한다.

**사용자 스토리**
> As a 인증된 회원, I want to 제목과 시작일·종료일을 지정해 할일을 만들고 싶다, So that 기한이 명확한 과업을 관리할 수 있다.

**수용 기준**

| # | Given | When | Then |
|---|-------|------|------|
| AC-03-1 | 유효한 액세스 토큰 보유 | 제목·시작일·종료일(종료일 >= 시작일)을 입력하고 생성 요청 | 할일이 저장되고 생성된 할일 객체(상태 포함)를 반환한다 |
| AC-03-2 | 유효한 액세스 토큰 보유 | 제목 또는 시작일 또는 종료일 중 하나라도 누락 | TODO_001 오류를 반환한다 |
| AC-03-3 | 유효한 액세스 토큰 보유 | 종료일 < 시작일 | TODO_002 오류를 반환한다 |
| AC-03-4 | 액세스 토큰 없음 또는 만료 | 생성 요청 | AUTH_001 오류를 반환한다 |

### UC-04: 할일 목록 조회

**설명**
인증된 회원이 자신의 할일을 상태 필터·정렬·페이지네이션을 적용해 조회한다. 각 항목의 상태는 서버 UTC 기준으로 동적 계산된다.

**사용자 스토리**
> As a 인증된 회원, I want to 내 할일을 상태별로 필터링하고 마감일 순으로 보고 싶다, So that 현재 진행 상황을 한눈에 파악할 수 있다.

**수용 기준**

| # | Given | When | Then |
|---|-------|------|------|
| AC-04-1 | 인증된 회원, 본인 할일 3건 존재 | 필터·정렬 없이 목록 조회 | 본인 할일만 종료일 오름차순으로 최대 20건 반환한다 |
| AC-04-2 | 인증된 회원 | 상태 필터 `기한초과` 지정하여 조회 | 기한초과 상태 항목만 반환한다 |
| AC-04-3 | 인증된 회원 | 정렬 기준 `createdAt DESC` 지정 | 생성일 내림차순으로 정렬된 목록을 반환한다 |
| AC-04-4 | 인증된 회원, 50건 존재 | `page=2, limit=20` 지정 | 21~40번째 항목을 반환한다 |
| AC-04-5 | 인증된 회원 | `limit=101` 지정 | 최대 100건으로 제한하여 반환한다 |

**상태 산출 규칙 (서버 UTC 기준)**

| 우선순위 | 조건 | 상태 |
|----------|------|------|
| 1 | done=true AND 현재 <= dueDate | 성공완료 (Completed) |
| 2 | done=true AND 현재 > dueDate | 지연완료 (LateCompleted) |
| 3 | done=false AND 현재 < startDate | 시작전 (Pending) |
| 4 | done=false AND startDate <= 현재 <= dueDate | 진행중 (InProgress) |
| 5 | done=false AND 현재 > dueDate | 기한초과 (Overdue) |

### UC-05: 할일 완료 처리 / 완료 취소

**설명**
인증된 회원이 특정 할일의 `done` 값을 토글(true <-> false)한다. 상태는 토글 후 즉시 재계산된다.

**사용자 스토리**
> As a 인증된 회원, I want to 할일을 완료 처리하거나 완료를 취소하고 싶다, So that 실제 진행 상황을 정확하게 반영할 수 있다.

**수용 기준**

| # | Given | When | Then |
|---|-------|------|------|
| AC-05-1 | done=false인 본인 할일, 현재 <= dueDate | 완료 처리 요청 | done=true로 변경, 상태=성공완료로 응답한다 |
| AC-05-2 | done=true인 본인 할일 | 완료 취소 요청 | done=false로 변경, 현재 날짜 기준 상태를 재계산해 응답한다 |
| AC-05-3 | 타인 소유 할일 | 완료 처리 요청 | TODO_003 오류를 반환한다 |
| AC-05-4 | 존재하지 않는 할일 ID | 완료 처리 요청 | TODO_004 오류를 반환한다 |

## 7. 데이터베이스 스키마 개요

PostgreSQL 기준

### users 테이블

```sql
CREATE TABLE users (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email       VARCHAR(255) NOT NULL UNIQUE,
  password    VARCHAR(255) NOT NULL,
  name        VARCHAR(100) NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

비고: password 컬럼은 bcrypt 해시 값을 저장한다.

### refresh_tokens 테이블

```sql
CREATE TABLE refresh_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token       VARCHAR(512) NOT NULL UNIQUE,
  expires_at  TIMESTAMPTZ NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### todos 테이블

```sql
CREATE TABLE todos (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  title        VARCHAR(255) NOT NULL,
  description  TEXT,
  start_date   DATE NOT NULL,
  due_date     DATE NOT NULL,
  done         BOOLEAN NOT NULL DEFAULT FALSE,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_todos_owner_id        ON todos(owner_id);
CREATE INDEX idx_todos_owner_due_date  ON todos(owner_id, due_date ASC);
```

**설계 메모**
- start_date, due_date는 UTC 기준 YYYY-MM-DD 날짜 값이다.
- status는 저장하지 않으며 조회 시 서버에서 동적 계산한다 (현재 UTC 날짜 + done + start_date/due_date 기반).
- ON DELETE CASCADE: 회원 삭제 시 할일·리프레시 토큰 자동 삭제 (NFR-004 대응).
- due_date >= start_date 제약은 애플리케이션 레이어에서 검증한다 (BR-T02).
